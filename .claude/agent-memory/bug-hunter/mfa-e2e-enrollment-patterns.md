Verified 2026-07-13 while writing `tests/e2e/mfa-enrollment.spec.ts` (THER-016 regression coverage).

**Two independent MFA login mechanisms coexist — use the right one for e2e.**
1. **Live path**: `authenticate()` in `src/app/actions/auth.ts` checks `lookupUser.mfaEnabled` *before* calling `signIn()`, mints a Redis-backed challenge via `createMfaChallenge()` (`src/lib/mfa-challenge.ts`), and passes `redirectTo: /mfa/verify?challenge=...` straight into `signIn()`. So a fresh credentials login for an MFA-enabled user lands on **`/mfa/verify?challenge=...`** (heading "Two-Factor Authentication"), not `/verify-2fa`. This page auto-POSTs `/api/auth/mfa/send` on mount and submits to `/api/auth/mfa/verify` (both under `src/app/api/auth/mfa/`).
2. **Step-up gate**: `proxy.ts` separately redirects any `/dashboard/*` or `/onboarding/*` request to **`/verify-2fa`** (heading "Verify it's you") when the JWT already has `mfaEnabled:true` but `mfaVerified` isn't true yet — this fires for an *existing* session whose MFA state changed underneath it, not for a fresh login. Its actions live in `src/app/actions/verify-mfa.ts`.
Both call the same underlying `verifyUserMfaCode`/`sendLoginMfaCode` from `src/app/actions/mfa.ts` and use the same `OtpInput` component (6 boxes, `aria-label="Digit N"`, auto-submits via `onComplete` when the last digit lands). Don't assume the wrong one — a `page.waitForURL('**/verify-2fa**')` after a normal login times out silently because the real redirect target is `/mfa/verify`.

**An org-less admin gets blocked by `OrganizationActivationModal`, not by proxy.ts.** Unlike the worker portal (which proxy.ts force-redirects to `/onboarding-worker` when `!token.organizationId`), the admin/`/dashboard` context has no such middleware-level redirect. Instead `src/app/dashboard/(main)/layout.tsx` renders `OrganizationActivationModal` whenever `!organizationId` — a welcome-mode Dialog that suppresses outside-click/Escape dismissal and auto-`router.push('/onboarding')`s after 60s. A Playwright test that seeds an org-less admin/owner (mirroring `onboarding-wizard.spec.ts`'s `seedUnboardedOwner` pattern) and then visits any other `/dashboard/*` page will silently time out clicking real page content, because the modal overlay blocks interaction until it eventually navigates away — the failure surfaces as a `getByRole` timeout with the final page snapshot showing `/onboarding/step1`, which is easy to misread as a routing bug. **Fix: seed a minimal Organization + Facility row and attach the test user to it** whenever the test needs a normal, uninterrupted `/dashboard/*` page (see `mfa-enrollment.spec.ts`'s `seedMfaTestUser`). `users`→`organizations`/`facilities` are `onDelete: Restrict`, so cleanup must delete the user row before the org/facility rows.

**There is only ever one `mfa_factors` row per user (until a second factor type is added).** `requestMfaSetup()` (enrollment), `sendLoginMfaCode()` (login step-up), and `sendDisableMfaCode()` all call `prisma.mfaFactor.update()` on the *same* row, overwriting `secret` in place — they never insert a new row. This means a naive "poll DB until `mfa_factors.secret` is non-null" helper will happily return the **stale** value from a *previous* send if called right after triggering a new one (race condition — the row was already non-null before the fresh send committed). Poll for a secret that differs from the last-known value instead (pass the previous secret as an exclusion param), not just for non-null. See `pollForEmailFactorSecret(userId, excludeSecret, timeoutMs)` in `mfa-enrollment.spec.ts`.

**TwoFactorAuthTab lives at `/dashboard/profile`'s "TWO FACTOR AUTH (2FA)" tab, not `/dashboard/settings`.** `src/components/dashboard/ProfileForm.tsx` renders it as a plain state-driven button-tab (not Radix Tabs, so no [[onboarding-invite-settings-phase-tests]]-style `getByLabel` gotcha) — `page.getByRole('button', {name: /two factor auth/i})` works directly. A separate, unused `src/components/auth/MfaSettings.tsx` component exists in the repo but is never imported anywhere — don't confuse the two.

**OTP decryption for e2e**: `src/lib/mfa.ts` encrypts OTP payloads as base64(iv(12) + authTag(16) + AES-256-GCM ciphertext) with key = sha256(NEXTAUTH_SECRET), plaintext = `JSON.stringify({code, createdAt})`. Decrypting in a spec requires `NEXTAUTH_SECRET` to be exported in the shell running Playwright (not just present in `.env`) — see [[e2e-local-auth-url-env-trap]] for the general pattern of the local runbook needing the full CI env block exported explicitly.
