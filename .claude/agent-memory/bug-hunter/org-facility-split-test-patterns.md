Verified 2026-07-02 on branch `rbac` (dev merged in, commit 62874ec): the Organization→Facility split moved `timezone`, `address`, `city`, `state`, `country`, `zipCode`, `phone`, `licenseNumber`, `staffCount`, `programServices`, `complianceDocumentUrl/Name` off `Organization` onto a new `Facility` model (one facility per org, created alongside it). `src/lib/reminders/us-state-timezone.ts` derives a facility's timezone from its US state at creation (`deriveTimezoneFromState`), falling back to `DEFAULT_TZ` (`America/New_York`).

**Regression-trap gotcha — DEFAULT_TZ equals the most common real timezone.** `DEFAULT_TZ` is `America/New_York`, and most existing test fixtures for `src/lib/reminders/sweep.test.ts` and `compliance.test.ts` used `America/New_York` (or `null`) as their "normal" facility timezone. That fixture choice means a test asserting the day-math result would **still pass** even if the code regressed to reading a removed `organization.timezone` field (which is `undefined` on the mock, so `?? DEFAULT_TZ` silently kicks in and happens to match). **Fix:** any test guarding "reads timezone from X" must use a timezone genuinely different from `DEFAULT_TZ` (e.g. `America/Los_Angeles`, UTC-7 vs UTC-4) and pick a `dueAt`/`now` pair whose *local calendar date* actually differs between the two zones — not just any different string. Verify the day-math manually (`startOfDayInTz` local-date crossing) before asserting; a naive choice can accidentally produce the same day count in both zones. Also add a structural assertion on the exact prisma `select` shape (`select.user.select.facility` present, `select.user.select.organization` undefined) as a second, independent guard. See `src/lib/reminders/sweep.test.ts` and `compliance.test.ts` for the applied pattern. Related: [[reminders-test-patterns]].

**Billing seat/plan-limit checks read `organization.facilities[0].staffCount`, not `organization.staffCount`.** `src/app/api/billing/subscription/checkout/route.ts` and `src/app/actions/invite.ts` (seat counting) both read staff count via the Facility relation. A reversion to `organization.staffCount` reads `undefined`, `parseInt(undefined ?? '0')` → `0`, and the plan-restriction guard silently stops firing (any org can "downgrade" past its staff count). Regression test: set a facility staffCount that exceeds the plan's `staffMax` and assert a 422/rejection — the pre-existing `checkout/route.test.ts` had no such test until this session; it only exercised the double-charge (THER-001) guard.

**vi.clearAllMocks() does not clear a previously-set mockResolvedValue.** In `organization.test.ts`, an earlier `describe` block called `prisma.organization.findFirst.mockResolvedValue(existingOrgObject)`. A later `describe` block for `createOrganization` only called `vi.clearAllMocks()` in its `beforeEach` — that clears call history but **not** the implementation `mockResolvedValue` installed (only `vi.resetAllMocks()`/`mockReset()` does that). The leftover truthy resolution made `createOrganization` take the "duplicate org name" early-return path, so the transaction (and all its assertions) silently never ran. Fix: any test file with multiple `describe` blocks sharing one mocked prisma method across blocks must explicitly re-set that method's resolved value in each block's own `beforeEach`, not rely on `clearAllMocks` to reset it.
