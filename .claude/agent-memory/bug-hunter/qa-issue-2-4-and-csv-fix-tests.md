Five QA-report fixes (`qa-reports/phase1-foundation.md` ISSUES 1-5) landed on branch `rbac`: CSV import skip-row reporting + role pre-fill, removed-staff session kill (ISSUE 2), dual-cookie sibling-clear on login (ISSUE 4), invite-email role labels (ISSUE 5). New/extended test files: `src/lib/staff-csv.test.ts`, `src/lib/email.test.ts`, `src/lib/create-auth-instance.test.ts`, `src/lib/auth/session-cookies.test.ts` (new), `src/app/actions/staff.test.ts`, `src/app/actions/auth.test.ts`, plus new e2e specs `tests/e2e/rbac-dual-cookie-login.spec.ts` and `tests/e2e/rbac-removed-staff-login.spec.ts`.

**Why:** these gotchas cost real debugging time and will recur on any future e2e spec touching /dashboard or org-less admin/worker states.

**How to apply:**
- **`DashboardEmptyState` is a click-blocking trap for freshly-seeded e2e orgs.** Any brand-new org (0 courses) auto-opens a fixed `z-50` full-screen "getting started" overlay on `/dashboard` (`src/components/dashboard/DashboardEmptyState.tsx`) that intercepts ALL clicks, including the fixed sidebar. Its close button is icon-only with NO accessible name (`getByRole('button', {name: 'Close'})` will NOT match it — that only matches the separate `WorkerWelcomeModal` on `/worker`). Dismiss via `page.locator('div.fixed.inset-0.z-50 button').first().click()` (best-effort, wrapped in a visibility check) before interacting with anything on `/dashboard` for a fresh-seed test user. Existing specs (`manage-learn-switcher.spec.ts` etc.) don't hit this because they reuse a long-lived seeded fixture account (`admin@test.com`) that already has course data.
- **Org-less OWNER is modal-gated, not route-gated.** `authenticate()`'s redirectTo is ALWAYS `/dashboard` for admin-tier roles regardless of org status — there is no server redirect to `/onboarding`. `OrganizationActivationModal` (rendered in `dashboard/(main)/layout.tsx`, gated on `!organizationId`) instead shows a welcome dialog ON `/dashboard` with an "Activate your account" button that does `router.push('/onboarding')` (or a 60s unattended auto-redirect). Don't assert `waitForURL('**/onboarding**')` immediately after an org-less owner login — assert `/dashboard` + the modal's welcome copy, then click through if you need to verify `/onboarding` is reachable.
- **Org-less WORKER *is* route-gated (`proxy.ts`), but the FIRST client-side landing on `/worker` right after `signIn()` can visibly "stick" without the `/onboarding-worker` redirect firing — even after 45s.** Verified via raw `curl` with a real session cookie that the server-side 307 (`/worker` → `/onboarding-worker`) fires unconditionally and fast; the issue is specific to the client-side/soft navigation Next.js does immediately after a Server Action's `signIn(..., {redirectTo})`. Fix: after the initial `waitForURL(/\/(worker|onboarding-worker)/)`, force a hard `page.goto('/worker')` — this reliably reproduces the proxy redirect within ~2s. Don't burn time raising the timeout further; it's not a compile-cold-start issue, it's a soft-nav quirk.
- **The `signIn` callback's `cookies()` mutation (ISSUE 4's cookie-clear) DOES attach to the response at runtime** — confirmed via the FIRST-PRIORITY e2e assertion. The only false failure encountered was the webServer cold-compiling on the very first hit of a freshly-spawned Playwright-managed dev server (exceeded a 15s `waitForURL`); re-running against an already-warm, manually-started `npm run dev -- -p 3005` fixed it instantly. When debugging a suspicious first-test failure, always check whether the webServer was cold before treating it as a product bug.
- **`mfa-enrollment.spec.ts`'s second test ("step-up: the next login is redirected to /mfa/verify...") is a known pre-existing flake**, reproduced identically on a `git stash`-reverted baseline with none of the ISSUE 1-5 fixes applied. Not caused by this changeset — don't waste time re-investigating it against these fixes specifically.
- **`NEXTAUTH_SECRET` must be exported into the shell running `npx playwright test`** for `mfa-enrollment.spec.ts` to decrypt OTPs (`set -a; source .env; set +a` before the command) — the dev server loads `.env` itself, but a separate `playwright test` process does not inherit it automatically.
