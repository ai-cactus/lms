Verified 2026-07-06 on branch `rbac`: the single `worker` DB role was replaced by 8 job-specific `category:'worker'` roles (`psychiatrist_prescriber`, `nurse`, `therapist_clinician`, `case_manager`, `behavioral_health_technician`, `peer_support_specialist`, `front_desk_admin`, `facilities_support`), all sharing one identical permission ceiling. `Role` (`permissions.ts`) gained a `category: 'manager' | 'worker'` field. `ALL_ROLES` is now 13 entries (5 manager + 8 worker). `DEFAULT_SELF_SERVE_WORKER_ROLE = 'front_desk_admin'` is the fallback used wherever a worker account is created without an explicit job category (self-signup, invalid/stale invite role, CSV bulk-import default, onboarding step-4 bulk invites).

**Gotcha — a retired/unrecognized role string doesn't fail gracefully in `dbRoleToRoleKey`/`can()`; it throws.** `dbRoleToRoleKey(role)` is a plain object lookup (`DB_ROLE_TO_ROLE_KEY[role]`) with no fallback — an unrecognized value (e.g. the retired literal `'worker'` used as a stale test fixture, or a real stale JWT session minted before this migration) returns `undefined`, and the immediate `can(undefined, permission)` call in both `src/lib/rbac/authorize.ts` and `updateFacility()` in `src/app/actions/organization.ts` then does `roles[undefined].permissions` → unhandled `TypeError`, not a clean 401/403. `authorize.ts` has no try/catch around this, so it would surface as a raw exception (likely a 500) in any API route, not a graceful "Forbidden". This was caught here only because a test fixture (`makeSession('worker')`) hit it — worth flagging to code-ninja as a defensive-coding gap if a user could plausibly present a session with a stale/invalid role (e.g. a long-lived JWT signed before this deploy). Fixed in tests by using a real DB role literal (e.g. `'nurse'`) in fixtures instead of the retired `'worker'`/`'admin'` strings; the underlying `can()`/`dbRoleToRoleKey` robustness gap itself was left unfixed as a suspected product issue (not this agent's to fix).

**`GRANTABLE_ROLES` lists admin roles before worker roles, so an inviter's default role-picker selection is an admin role, not a worker role.** `src/components/dashboard/staff/InviteStaffModal.tsx` defaults `selectedRole` to `grantableRoles[0] ?? DEFAULT_SELF_SERVE_WORKER_ROLE`. For an `owner`/`supervisor` inviter, `GRANTABLE_ROLES.owner[0]` is `'supervisor'`; for an `hr` inviter it's `'hr'` itself (self-grant). So the modal's default invite role is currently an elevated admin role for every admin-category inviter, not the common-case worker role the original test comment assumed ("most invites are workers"). Confirmed via `git log -p` that this dynamic-default logic (`grantableRoles[0] ?? ...`) is itself part of this RBAC change (previously a hardcoded `useState<Role>('worker')`). Not flagged as a bug since it's an intentional new pattern, but worth a product/UX gut-check if surfaced again: defaulting an owner's invite modal to "supervisor" could be a silent privilege-escalation-by-default risk if an admin clicks through without changing the selector.

**Test fixtures using literal `'worker'` for a "worker DB role" are NOT necessarily broken — check whether the value flows through `can()`/`dbRoleToRoleKey`.** Many pre-existing test files (`create-org.test.ts`, `offering.test.ts`, `recipients.test.ts`) used `role: 'worker'` purely as a generic "some non-admin caller" fixture, checked only via `isAdminRole()` (which returns `false` for `'worker'` both before and after the split — it was never in `ADMIN_ROLES`). These continued to pass at runtime with zero code changes; only `tsc` on the *typed* call sites (function params literally typed `UserRole`) caught the drift. Don't assume every `'worker'` grep hit is a live bug — trace whether it reaches `dbRoleToRoleKey`/`can()` (crashes, per above) or a DB enum insert (fails, since `'worker'` is no longer a valid `UserRole` value) before deciding it needs a behavioral fix vs. a cosmetic one.

**`verification_tokens.role` and `StaffEntry.role` (CSV import) are untyped `String`/coarse tokens, not the `UserRole` enum** — so literal `'worker'`/`'admin'` values compile fine and don't crash there; they get resolved downstream (`/api/auth/verify` validates against `ALL_ROLES` and falls back to `DEFAULT_SELF_SERVE_WORKER_ROLE`; `enrollUsers` CSV import maps `'admin'→'supervisor'`, everything else→`DEFAULT_SELF_SERVE_WORKER_ROLE`). Regression tests for these paths must assert the *resolved* DB role (e.g. `'front_desk_admin'`), not echo the coarse input token back.

**e2e specs under `tests/e2e/` seed users via raw `INSERT INTO users (..., role, ...) VALUES (..., $n::"UserRole", ...)`** — a literal `'worker'` there is a real Postgres enum cast and WILL fail at runtime post-migration (not just a stale assertion). Found and fixed in `rbac-roles.spec.ts`, `rbac-facility-tab.spec.ts` (unused, only the type union was stale), `rbac-invite-roles.spec.ts` (role-selector option-name assertions used a generic `/worker/i` regex — now meaningless since none of the 8 new roles' displayNames contain the word "worker", e.g. "Nurse", "Case Manager"). Also found in `signup-email-verification.spec.ts`: an AC-5 test asserted a *legacy* token role (`'admin'`) round-trips unchanged through `/api/auth/verify` — that's now actively wrong, since verify.ts's fallback logic (added this migration) maps any role not in `ALL_ROLES` to `DEFAULT_SELF_SERVE_WORKER_ROLE`; rewrote as two tests: real founder signup (`role:'owner'`) round-trips correctly, and a legacy/stale token role safely falls back rather than persisting an invalid enum value. None of these e2e specs were executed live (would need the app + migrated DB running); fixed for compile/logical correctness against current source only.
