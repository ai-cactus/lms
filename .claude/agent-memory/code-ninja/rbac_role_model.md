RBAC model (rolled out on the `rbac` branch):

- **DB enum `UserRole`** (snake_case), 13 values, replacing the old `admin | worker`. Five **manager**-category roles: `owner`, `supervisor`, `hr`, `clinical_director`, `finance`. Eight **worker**-category roles (all sharing ONE identical permission ceiling — they differ only by category + display name): `psychiatrist_prescriber`, `nurse`, `therapist_clinician`, `case_manager`, `behavioral_health_technician`, `peer_support_specialist`, `front_desk_admin`, `facilities_support`. The single old `worker` role was removed. Migration `20260701120000_rbac_roles` maps `admin -> supervisor`, `worker -> front_desk_admin`, `ELSE -> front_desk_admin`; neither `users.role` nor `invites.role` carries a DB DEFAULT anymore — every create must pass an explicit role. `scripts/backfill-roles.js` then promotes each org's earliest supervisor to `owner`.
- **`src/lib/rbac/permissions.ts`** is the authoritative permission registry. `Role` interface has a `category: 'manager' | 'worker'` field. The 8 worker roles all reuse a single shared `const workerPermissions` array (defined once — never duplicate the literal 8×). RoleKeys are camelCase (multi-word DB values camelCased, e.g. `clinical_director→clinicalDirector`, `front_desk_admin→frontDeskAdmin`). `supervisor` = `everythingExceptBilling`; `owner` = full `everything`. So `billing.*` holders = **owner + finance only**.
- **`src/lib/rbac/role-utils.ts`** is the ONLY place snake↔camel conversion happens (`dbRoleToRoleKey`, exhaustive `DB_ROLE_TO_ROLE_KEY`), plus `ADMIN_ROLES`/`WORKER_ROLES`/`ALL_ROLES` (typed `readonly Role[]`; spread as `{ in: [...WORKER_ROLES] }` in Prisma filters), `GRANTABLE_ROLES`, `getRoleDisplayName`, and predicates `isAdminRole`/`isWorkerRole`. Client-safe. Also exports **`DEFAULT_SELF_SERVE_WORKER_ROLE = 'front_desk_admin'`** — the fallback worker role used everywhere the system must create a worker without an explicit category (self-signup, OAuth-no-invite, invalid-invite-role fallbacks in `auth/verify` + `create-auth-instance`, join-by-code, onboarding step-4 invites, course-assignment auto-create, CSV import non-admin). Low-stakes label (all worker roles share perms). Mirrored in the migration SQL header (kept in sync manually — SQL can't import TS). NOTE: `isAdminRole`/`isWorkerRole` test DB roles — they do NOT recognize the legacy CSV/routing token `'admin'`, so CSV-import role parsing must use a literal `entry.role === 'admin'` check.
- **`src/lib/rbac/authorize.ts`** — `authorize(permission)` route guard returning `{ ok, ctx }` or `{ ok:false, response }`.

Key decisions baked in: `owner` is NON-grantable (absent from every `GRANTABLE_ROLES` array) and rejected as a target in `updateStaffDetails` (existing owner keeping role during a name edit is allowed). owner & supervisor may grant supervisor/hr/clinical_director/finance + all 8 worker roles; HR may grant hr/clinical_director/finance + all 8 worker roles (NOT supervisor/owner); clinical_director, finance, and all 8 worker roles grant nothing. Every role except `owner` consumes a plan seat (`role: { not: 'owner' }`). New org founder (OAuth-no-invite, credentials onboarding, org creation) becomes `owner`. `InviteStaffModal` sources its role `<select>` dynamically from `GRANTABLE_ROLES[inviterRole]` + `getRoleDisplayName`. `EditStaffModal` renders NO role picker (role state passes through unchanged). See [[auth-instance-vs-role]].

Legacy `'admin'|'worker'` binary is a SEPARATE concept from DB roles — it's a routing/UI-context token (proxy getContext, cookiePrefix, signup/onboarding role-selection UI, notification audience, CoursePreview mode, mfa-challenge, `src/types/enrollment.ts` StaffEntry.role). Leave those `'worker'` literals alone; they are not RBAC enum values. Superadmin `SystemUsersClient.tsx` still has a hardcoded `<option value="worker">` + `u.role === 'worker'` check (loosely typed, not compile-flagged) — left untouched pending a product decision on the system panel.
