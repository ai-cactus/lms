**Phase 4 LOCAL revalidation, completed (2026-07-17, `qa-reports/phase4-billing-audit-local-revalidation.md`), real Stripe test mode + live ngrok webhook tunnel — verdict PASS WITH RISKS.** After the `.env.local` dummy-key blocker was fixed (see [[local-dev-env-setup]]) and a stale `unstable_cache` price-fetch entry was cleared (required a second `next dev` restart — deleting the on-disk `.next/dev/cache/fetch-cache/` file alone was NOT enough; an in-process cache layer also held the pre-fix empty result), 4 of 5 Phase-4 fixes were confirmed FIXED end-to-end: (1) price display exactly matches Stripe/charged amount across Monthly/Quarterly/Yearly and both tiers; (2) plan-swap confirmation dialog + Monthly-cycle-toggle correctly persists (never resets to Yearly) on revisit; (4) pause blocks assign-page (redirect) + course-create (redirect) + audit-reports (modal), server action `enrollUsers` has its own `hasActiveBilling()` throw as defense-in-depth, and a paused-org worker still fully accesses already-assigned courses; (5) Overview reflects active state immediately after resume, no manual reload needed. Zero webhook signature failures, exactly 1 Stripe subscription + 1 invoice throughout despite a full subscribe→schedule-cycle→cancel-schedule→upgrade-dialog→pause→resume→cancel→resume sequence (no recurrence of the historical duplicate-subscription bug).

**NEW confirmed defect — Issue 3's `immediate_prorate` upgrade path is mathematically unreachable for MONTHLY-cycle subscriptions.** `isLessThanOneMonthRemaining()` in `src/lib/billing-plan-change.ts` compares `now` against `currentPeriodEnd - 1 calendar month`; for a monthly cycle, `currentPeriodEnd` is by definition `subscriptionStart + 1 month`, so that boundary collapses to `subscriptionStart` itself — meaning ANY upgrade attempt after the literal instant of subscribing (i.e. every real user action) classifies as `scheduled`, never `immediate_prorate`. Live-reproduced: upgrading Starter→Professional ~72 minutes after subscribing to Starter/Monthly still showed the "no charge today, starts at period end" dialog language, not "charge you $X now". The existing unit tests (`billing-plan-change.test.ts`) don't catch this because they pass `now` and `currentPeriodEnd` as independent free variables (e.g. a test using `now=T` and `currentPeriodEnd=T+1month` to hit "exactly the boundary") — a combination that can never arise from a real monthly subscription where `periodEnd` is derived from `now`'s own history, not chosen independently. Confirmed via the same math that Quarterly/Yearly cycles do NOT have this problem (their periods are >1 month long, so there's a genuine early-period window where `immediate_prorate` is reachable) — not independently live-verified though. Severity: High — silently defeats the entire "charge immediately on upgrade" business intent for the most common (monthly) cycle.

**NEW confirmed defect — Audit Reports export jobs complete successfully server-side but the browser never detects completion or auto-downloads.** Every export (worker/course/org, with or without a date-range filter) reaches `jobs.status='completed'`, `progress=100`, `message='Report Ready'` in the DB within ~2 seconds, and the PDF is always correctly fetchable via `GET /api/auditor/export/{jobId}/download?format=pdf` (verified with `pdftotext` — data is complete/accurate in every case, including the date-range-filtered case). But the client-side polling in `ExportJobsProvider`/`useJobStatus` (`GET /api/auditor/export/{jobId}/status`, meant to fire every 1.5s) never actually fires — confirmed via `playwright-cli requests` showing zero such GET calls over 20+s of observation per job, even though manually calling that same endpoint via `page.evaluate(fetch(...))` works fine. User-visible consequences: the toast is stuck reading "Preparing X…" forever (contradicts the Export tab's own "downloads automatically" copy), and — worse — the org-level "Export PDF" button on the Export tab becomes **permanently stuck disabled** ("Generating… 0%") after the very first click, because the never-resolving job is persisted to `localStorage['auditReportExportJobs']`, blocking ALL further org-level exports via that button until a user manually clears browser storage. Severity: High (blocks a promised feature entirely from a real user's perspective, though the underlying data pipeline is sound). To verify a job actually completed despite this, query `jobs` table directly by `type='AUDITOR_PACK_EXPORT'` or call the `/status`/`/download` endpoints via `page.evaluate` — don't trust the toast/button UI state.

**Gotcha confirmed again — `unstable_cache`'s dev-mode persistence survives a `next dev` restart via BOTH an on-disk file (`.next/dev/cache/fetch-cache/<hash>`) AND a separate in-process layer.** Deleting the disk file is not sufficient to force a fresh fetch if the in-memory layer already holds a (possibly stale/wrong) cached result from before your fix landed — you need a genuine fresh process restart (not just deleting the cache file) to be sure. See [[local-dev-env-setup]].

**NEW — Phase 4 billing/audit QA run (2026-07-16, staging, `qa-reports/phase4-billing-audit.md`), four open defects found, none of them a recurrence of the fixed THER-001/THER-010 duplicate-subscription bug (confirmed only one live Stripe subscription existed throughout):**

1. **No confirmation step exists before a plan swap, at all.** `SubscriptionTab.tsx`'s `handleSelectPlan` calls `POST /api/billing/subscription/checkout` directly on the "Subscribe" button click for every plan card — there is no confirm dialog anywhere in that code path. Confirmed live: clicking "Subscribe" on Professional while on Starter instantly swapped with zero dialog.
2. **The "Billing cycle" toggle (Monthly/Quarterly/Yearly) on the Subscription tab is local component state that resets to "Yearly" on every mount/reload — it is NOT derived from the org's actual current billing cycle.** Combined with bug #1, an admin who is genuinely on a Monthly plan, revisits the Subscription tab, and clicks Subscribe on any plan is silently switched to a **yearly** commitment they never explicitly chose. Confirmed live: Starter/Monthly → Professional/**Yearly** in one click.
3. **Plan-swap proration is never actually invoiced.** The checkout route calls `stripe.subscriptions.update(..., { proration_behavior: 'create_prorations' })` with no `proration_behavior: 'always_invoice'` and no follow-up `stripe.invoices.create()` — Stripe's default behavior attaches proration items to the customer's *next* invoice rather than billing immediately, contradicting the route's own code comment ("bills/credits the difference immediately"). Confirmed live via Stripe's Customer Portal: after a Starter/Monthly → Professional/Yearly swap, no new invoice appeared; instead a "next estimated payment: US$3,476.00" was deferred a full year to the new yearly renewal date. The org gets upgraded features immediately with the true cost uncollected for up to a year.
4. **The Stripe test-mode price actually charged does not match the price advertised in the app's own plan cards.** App showed Starter at "$99/mo"; Stripe Checkout displayed and charged **$124.00/mo** (confirmed via the Checkout total, the `deferred_intent[amount]=12400` Stripe Elements network request, and the resulting paid invoice). Same class of mismatch observed on the Professional/Yearly swap ($1,800/yr actual vs the ~$1,341/yr implied by the app's own `$149 × 12 × 0.75` yearly-discount math). Likely cause: the Stripe test-mode price object IDs referenced by `STARTER_MONTHLY_PRICE_ID`/`PROFESSIONAL_MONTHLY_PRICE_ID` point to price objects with different amounts than the static `monthlyPrice` figures hardcoded in `src/lib/billing-plans.ts`. **Always compare the Checkout page's displayed total against the app's own advertised price before assuming they match — do not assume the config and the live Stripe test price are in sync.**
5. **Course *assignment* (assigning an existing/video-catalog course to a worker) has NO billing gate at all**, unlike course *creation* (`courses/create/page.tsx`, gated) and Audit Reports (`AuditorBillingGateWrapper`, gated via `hasAuditorAccess`). Confirmed via code read (`training/courses/[id]/assign/page.tsx` and the underlying enrollment action never reference `hasActiveBilling()`) and live: while the org's subscription was fully **paused** (sitewide "Access is limited" banner visibly showing), navigating directly to a course's `/assign` page and submitting succeeded with a normal "Course Assigned Successfully" confirmation — no gate/warning of any kind. If a story needs "assignment is billing-gated" to be true, expect it to fail as of this date.
6. **Minor UX bug, not a correctness bug:** after a successful pause→resume ("Continue Plan"), the Overview/Subscription tab UI does not refresh to show the resumed state — it keeps showing "Paused" until a manual page reload, even though the resume API call itself succeeded (`200`). A second click on the stale button correctly 409s ("Subscription is not paused."). The underlying billing state is correct throughout; only the client-side re-render is stale. Use a `reload`/fresh navigation, not just the post-click snapshot, to verify true billing state after any pause/resume/cancel mutation.

**Cancel-flow re-verification (2026-07-16, still correct):** the "Cancel or Pause" confirmation modal, the "you keep access until [date]" copy, and the cancel-reason survey all still work as described below. **New confirmation: cancelled-plan re-cancel protection works at both UI and API layers** — once a cancellation is scheduled, the Subscription tab shows only "Resume subscription" (no Cancel button at all), direct navigation to `/dashboard/billing/cancel` redirects server-side back to `/dashboard/billing`, and a direct repeat `POST /api/billing/subscription/cancel` call correctly 409s (`"Subscription is already scheduled for cancellation."`).

**Audit-report billing gate (`hasAuditorAccess` on the `Organization` row, distinct from but kept in sync with `hasActiveBilling()`) re-verified working correctly across all transitions:** accessible immediately after subscribing, correctly gated (`"A plan is required for reports"` modal) while paused, and accessible again once resumed or while merely "scheduled to cancel but still within the paid period."

**FIXED as of 2026-07-02 (branch `fix/qa-report-001`, ticket THER-001/THER-010) — re-verified live with full evidence.** Switching plans while a subscription is active now calls `POST /api/billing/subscription/checkout`, which updates the EXISTING Stripe subscription in place (confirmed via Stripe API: same `sub_...` id before/after, price swapped, only one subscription ever exists) instead of opening a new Checkout session. No Checkout redirect occurs on a swap — the app returns `{"updated":true}` synchronously. "Current Plan" was stable across 5+ reloads on both Billing and Staff Management (no flip-flop). Only one invoice existed throughout (no duplicate full-price invoice). Full write-up: `qa-reports/feat-qa-002-validation.md` (Story 1). The historical bug description below (2026-07-01, pre-fix) is kept for context/regression-testing reference — don't treat it as current behavior.

**HISTORICAL — pre-fix bug, confirmed 2026-07-01, RESOLVED 2026-07-02 — "upgrading" a plan creates a SECOND independent Stripe subscription instead of modifying the existing one.** On `/dashboard/billing?tab=subscription`, clicking "Subscribe" on a higher-tier plan while already subscribed to a lower one opens a **brand-new Stripe Checkout Session** (fresh `cs_test_...` ID, requires re-entering card details from scratch — no saved-card reuse) rather than calling Stripe's subscription-update/proration API. Confirmed via Stripe's own Customer Portal showing a "Current subscriptions" (plural) section with BOTH the old and new plan simultaneously active, both billing on the same next date — i.e. the org would be double-charged (e.g. $99 + $149 = $248/mo) if left unresolved. Full writeup: `qa-reports/phase-8-billing.md` (ISSUE 1).

**Downstream consequences of the duplicate-subscription bug (also confirmed):**
- The in-app "Current Plan" display (`/dashboard/billing?tab=overview` AND `?tab=subscription`) becomes **non-deterministic** — it flip-flops between showing the old and new plan across page loads/reloads with zero user action in between. Don't trust a single observation of "Current Plan" as ground truth if more than one subscription might exist; always cross-check Stripe's Customer Portal directly.
- The in-app "Cancel or Pause" page (`/dashboard/billing/cancel`, reached via the Subscription tab's "Cancel or Pause" button) can reference a **stale/wrong subscription** (e.g. still saying "Cancel Starter Service" after Professional became current) — don't submit an action from this page without first confirming via Stripe's Customer Portal which subscription is actually current, to avoid further compounding the duplication.

**How to reach Stripe's own Customer Portal for ground-truth verification:** clicking "Add Payment Method" on the app's Payment Method tab actually opens the **full Stripe-hosted Customer Portal** (`billing.stripe.com/p/session/test_...`, "Test mode" badge), not a scoped payment-method-only view. This portal is the most reliable place to check actual subscription state (shows a "Current subscriptions" list with per-subscription Update/Cancel links, real subscription IDs like `sub_...`, and accurate next-billing dates) — more trustworthy than the app's own Overview/Subscription tabs when things look inconsistent.

**Verifying genuine Stripe TEST mode (not live) before entering any card data:** look for `cs_test_` in Checkout URLs, `test_` in Customer Portal session URLs, and an on-page "Sandbox" (Checkout) or "Test mode" (Customer Portal) badge. Stripe test cards (e.g. `4242 4242 4242 4242`) are technically rejected by Stripe itself if live-mode keys were somehow in play, providing a natural safety net — but always positively confirm the test-mode indicator rather than relying solely on being told it's test mode.

**Stripe Checkout page has an "I am an AI agent acting on behalf of someone else" disclosure checkbox** with an associated "Link CLI" flow (`which link-cli`, asks the buyer to install it so the AI agent never sees the buyer's real card). This checkbox and its hidden input have `tabindex="-1"` and sit outside the normal viewport — `playwright-cli click <ref>` on it will time out with "element is outside of the viewport"; work around with `playwright-cli eval "el => { el.click(); return el.checked; }" <ref>` to click it directly via JS. Only actually follow the Link CLI flow for REAL purchases with a real buyer's real card; for pre-authorized, publicly-documented Stripe test cards (like this QA scenario), checking the disclosure box honestly (transparency) is appropriate, but completing the Link CLI steps is not, since there's no real credential to protect.

**Stripe's embedded card element (both Checkout and Customer Portal "Add payment method") uses placeholder-based locators reliably:** `getByPlaceholder('1234 1234 1234 1234')` (card number), `getByPlaceholder('MM / YY')` (expiry), `getByPlaceholder('CVC')` (security code), `getByPlaceholder('Full name on card')` (Checkout only). Selecting "United States" from the "Country or region" combobox reveals a `getByRole('textbox', {name:'ZIP'})` (Checkout) or `'ZIP code'` (Customer Portal) field that isn't present for the default-detected country (was "Nigeria" in this environment — geo-IP based, not org-address-based).

**Standard Stripe cancel-subscription behavior (confirmed correct when reached via the Customer Portal):** cancellation is NOT immediate — access continues until the current billing period ends, with an explicit "If you cancel this subscription, it will still be available until the end of your billing period on [date]" message, followed by an optional reason-for-leaving survey (dropdown: "It's too expensive" / "I found an alternative" / "I no longer need it" / "Other reason"), then a final "Cancel subscription" confirm button. The org's own in-app pause/cancel page (`/dashboard/billing/cancel`) offers the same "still have access until [date]" pattern plus a distinct "Pause Instead" option (1/2/3 months) — but see the stale-reference bug above before trusting which subscription it targets.
