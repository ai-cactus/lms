Confirmed live end-to-end on branch `rbac` (`origin/dev` merged in) on 2026-07-09. Supersedes the old role-selection-based signup flow noted in [[local-dev-env-access]] and [[rbac-role-grant-matrix]] — **self-serve `/signup` no longer has a role-selection step at all**; it always mints role `owner` (hardcoded `persistedRole = 'owner'` in `src/app/actions/auth.ts`, with an explicit comment that self-serve always founds an org). Worker/manager accounts can only be created via invites or join-by-code.

**Signup → verify → onboarding route sequence:**
`/signup` (first/last name, email, password/confirm, terms — Radix checkbox needs a direct role-click, `fill`/native-input approaches don't register) → submit → `/verify-email` interstitial → mailed link is `/api/auth/verify?token=...` which 307s to `/verify?token=...` ("Welcome to Theraptly" page) → click "Verify Email Address" → POSTs and redirects to `/login?verified=true`. Logging in with no `organizationId` yet lands on `/dashboard` showing an **"Activate your account" modal** (not a hard redirect) — clicking it goes to `/onboarding/step1`.

**Onboarding wizard is 5 steps, all client-side-drafted to `localStorage['onboarding_data']` until the final submit:** step1 (org/legal details — "Skip for now" goes straight to `/dashboard` without creating an org!), step2 (credentialing + optional doc upload — HIPAA Compliance Confirmation combobox is required with no skip escape hatch, unlike step1), step3 (business type + program services, Radix multi-select popover), step4 "Invite your managers" (manager invites — role dropdown scoped to `MANAGER_INVITE_ROLES`: Supervisor/HR/Clinical Director/Finance — starts with exactly ONE empty email+role row, skippable), step5 "Invite your Workers/Staffs" (worker invites, skippable) → `completeOnboarding()` server action creates the `Organization` + `Facility` and sets the user's `organizationId`/`facilityId` → `/onboarding/complete` ("You're all Set!") → "Go to Dashboard" → clean `/dashboard`, activation modal gone, sidebar now shows a "SETTINGS" section (Settings + Billing links, owner-only).

**UPDATE 2026-07-09 (commit `ec669ba`), confirmed live via full retest:** Step 5 was rebuilt from an emails-only `TagInput` to the **same per-row email + role interface as step 4** — Email/Roles column header, one textbox + one role `Select` per row, `Remove row` (disabled when only one row), `Add team member`, `Import with .csv file instead`, `Download sample .csv template` (downloads `workers-template.csv` with an `email,role` header and worker-role-slug samples, e.g. `nurse`, `therapist_clinician`, `case_manager`). **Both step4 and step5 now start with exactly ONE empty row** (step4 previously seeded three; step5 was previously a bare tag input). Step5's role `Select` offers all 8 worker roles: Psychiatrist/Prescriber, Nurse, Therapist/Clinician, Case Manager, Behavioral Health Technician ("...Mental Health Associate"), Peer Support Specialist, Front Desk/Administrative Support, Facilities/Support Staff. A row with an email but no role selected blocks `Next` with an inline "Select a role" error and does not advance. Completing onboarding now mints an `Invite` DB row with the **exact chosen worker role** (verified via direct DB `SELECT` on `invites.role`) — workers are no longer blanket-assigned `front_desk_admin`. Full retest report: `qa-reports/owner-signup-onboarding-invite-rework.md`.

**Staff invites live in two equivalent places** using the same `InviteStaffModal` component + `createInvites` server action: (1) onboarding step4/5 (manager/worker invites during setup), and (2) post-onboarding at `/dashboard/settings` → "Users & Permissions" tab → "Invite user" button (also exposed at `/dashboard/staff` as an "Add Workers" button). The modal flow is: paste/type emails (or CSV import) → "Continue" → per-contact role assignment from a `groupRolesForSelect(inviterRole)`-scoped dropdown, grouped into "Managers" / "Workers / Learners" sections (owner sees all manager roles — Supervisor/HR/Clinical Director/Finance — plus all worker-category roles — Psychiatrist/Prescriber, Nurse, Therapist/Clinician, Case Manager, Behavioral Health Technician, Peer Support Specialist, Front Desk/Administrative Support, Facilities/Support Staff) → "Continue" sends the invite emails and shows an "Invite sent" success step. **Fixed 2026-07-09 (commit `6506d63`), confirmed live:** the singular/plural success copy now correctly reads "1 person has been invited." / "N people have been invited." (previously rendered "hasbeen"/"havebeen" with no space — a JSX-newline-collapse bug). Two unrelated, still-present, non-blocking React warnings surfaced during this flow: `Missing 'Description' or aria-describedby={undefined} for {DialogContent}` (a distinct a11y gap — the dialog now has a title but still lacks a description/aria-describedby) and `Select is changing from uncontrolled to controlled` on the per-contact role `Select` in the "Assign roles" step.

**`/join/[token]` three-way branch** (confirmed still correct post-`origin/dev`-merge, matching the fix already verified in [[rbac-join-token-critical-bug]]): pending+unexpired → real join form (org name, role, invitee email pre-filled/disabled); `status='accepted'` → friendly "This invite has already been used" page with a "Go to login" link (HTTP 200, not 404); anything else (unknown token, expired-and-never-accepted) → genuine Next.js `notFound()` (HTTP 404). Completing the join form POSTs `/api/invite/accept` and redirects to `/login?joined=true`; the new user's role/org/facility exactly match the invite.

**Settings (`/dashboard/settings`) is owner-only**, enforced server-side (not just nav-hidden) — a non-owner hitting the URL directly gets a real access-denied page render ("You don't have access to Settings"), confirmed live with a supervisor account. Three tabs: **Users & Permissions** (default; team table + "Invite user"), **Roles** (read-only reference matrix), **Facility** (editable org/facility name + type via `updateFacility` server action — form uses react-hook-form `isDirty` to gate the Save button, shows a "Facility updated." success alert, and the new values persist across a hard reload; also shows current subscription plan read-only with a "Manage in Billing" link, out of scope for Stripe-free QA runs). There is **no separate "profile" tab inside Settings** — personal profile editing is a different route entirely, linked from the header user-menu dropdown ("Profile" → `/profile`, not `/dashboard/settings`).

**FIXED 2026-07-09 (commit `6506d63`), confirmed live:** `InviteStaffModal`'s `DialogContent` previously used a plain `<h2>` instead of the shadcn/ui `DialogTitle`/`DialogHeader` primitives, triggering a Radix "`DialogContent` requires a `DialogTitle`..." console warning on every open. Now each of the three modal steps (`Invite New Staffs`, `Assign roles`, `Invite sent`) renders as an accessible-named `dialog` via real `DialogTitle` — zero occurrences of that warning across a full open→assign→success cycle in the retest.

See [[local-dev-env-setup]] for the MailHog-must-be-started-manually gotcha that blocks this entire flow if skipped, and [[signup-test-patterns]] for the Radix-checkbox and server-action-mocking gotchas that still apply to the new `/signup` form.

**UPDATE 2026-07-09 — Figma sidebar redesign + Manage/Learn mode switcher (commit `3fcae6e`), confirmed live.** The admin sidebar (`DashboardLayoutClient.tsx`) is now grouped: a **Manage\|Learn segmented switcher** at the very top (rendered only when `isAdminRole(role)` — i.e. every manager-category role, not just owner) → **MAIN MENU** (Dashboard, Documents, Courses, Status Tracker — Status Tracker itself is admin-only) → **PERFORMANCE** (Staff Management, Audit Reports, whole section admin-only) → **SETTINGS** (Settings link only if `role==='owner'`; Billing link only if `can(role,'billing.read')`; **Help Center always shown to every admin-tier role** at `/dashboard/help`). The worker sidebar (`WorkerDashboardLayout.tsx`) is simpler: the switcher renders there too but **only when the session's role is admin-tier** (i.e. a bridged manager, not a real worker) — MAIN MENU (Dashboard→`/worker`, Trainings, Certificates) + **HELP** (Help Center → `/worker/help`). A real worker-category account never sees any switcher at all.

**Manage/Learn bridge mechanics** (`src/app/actions/session-bridge.ts::enterLearnMode`): clicking "Learn" submits a hidden `<form action={enterLearnMode}>` (not a link) that re-verifies the caller's role from a **fresh DB read** (never trusts the JWT alone), then mints a **second, separate worker-instance session cookie** (`worker.session-token`, distinct from `admin.session-token`) for the same user — carrying their real (admin) role — and redirects to `/worker`. The worker auth instance's `sessionAllowedRoles` is `ALL_ROLES` (so this bridged token passes its own re-validation) while its **login-form** `allowedRoles` stays `WORKER_ROLES` — i.e. an admin can never *log in* directly to `/worker`, only arrive there via this bridge. Confirmed via a direct out-of-band POST to `/api/auth-worker/callback/credentials` with real admin credentials + a fetched CSRF token: rejected with `302 → /login?error=CredentialsSignin`, no session cookie set — good defense-in-depth beyond the app's own single `/login` role-router. Clicking "Manage" from `/worker` is a plain `<Link href="/dashboard">` (not another server action) — both cookies coexist; no logout occurs on either leg. Each successful "Learn" entry writes one `audit_logs` row (`action='auth.mode_switch'`, `metadata={"direction":"learn"}`, `actor_role`=the real admin role) — a reliable DB-side signal to confirm the bridge actually fired.

**Bridged manager's `/worker` view is genuinely self-scoped, not just UI-hidden** — both `src/app/worker/page.tsx` and `src/app/worker/trainings/page.tsx` query `prisma.enrollment.findMany({ where: { userId: session?.user?.id } })`, scoped to the *authenticated session's own* userId (which for a bridged session is the manager's own id, carried over from their admin JWT) — not by organization. Proved this live by SQL-seeding a course+enrollment for a *different* user (a real worker) in the same org, then confirming the bridged manager's Learn view still showed "No courses found" — see [[watch-through-gate-testing]]-style direct-SQL seeding pattern for how to insert a minimal `courses`+`enrollments` row pair (`status='draft'`→use `'published'`, `type='text'`) when the AI pipeline is unavailable.

**Gotcha — the dashboard "0 courses" empty-state modal and the worker-side `WorkerWelcomeModal` are two separate first-load modals, both real backdrop-trapping dialogs (not bugs).** The admin `/dashboard` shows a "Turn Your Healthcare Policies into Interactive Training in Minutes." modal for orgs with 0 courses (closable via a bare `button` — no accessible name in the snapshot, it's the 4th `button` role on the page, or match by proximity to that heading). The worker `/worker` dashboard separately shows "Your first training course awaits you" (`WorkerWelcomeModal`) for a learner with 0 enrollments, closable via its "Close" button. **Both correctly block pointer-events on everything behind them** (Playwright will report "subtree intercepts pointer events" if you try to click a sidebar link while one is open) — this is correct modal-trap behavior, not the previously-fixed join-by-code decorative-panel overlap bug; always close whichever modal is open before interacting with the sidebar.
