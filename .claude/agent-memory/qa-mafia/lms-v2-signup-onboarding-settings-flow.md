Confirmed live end-to-end on branch `rbac` (`origin/dev` merged in) on 2026-07-09. Supersedes the old role-selection-based signup flow noted in [[local-dev-env-access]] and [[rbac-role-grant-matrix]] — **self-serve `/signup` no longer has a role-selection step at all**; it always mints role `owner` (hardcoded `persistedRole = 'owner'` in `src/app/actions/auth.ts`, with an explicit comment that self-serve always founds an org). Worker/manager accounts can only be created via invites or join-by-code.

**Signup → verify → onboarding route sequence:**
`/signup` (first/last name, email, password/confirm, terms — Radix checkbox needs a direct role-click, `fill`/native-input approaches don't register) → submit → `/verify-email` interstitial → mailed link is `/api/auth/verify?token=...` which 307s to `/verify?token=...` ("Welcome to Theraptly" page) → click "Verify Email Address" → POSTs and redirects to `/login?verified=true`. Logging in with no `organizationId` yet lands on `/dashboard` showing an **"Activate your account" modal** (not a hard redirect) — clicking it goes to `/onboarding/step1`.

**Onboarding wizard is 5 steps, all client-side-drafted to `localStorage['onboarding_data']` until the final submit:** step1 (org/legal details — "Skip for now" goes straight to `/dashboard` without creating an org!), step2 (credentialing + optional doc upload), step3 (business type + program services, Radix multi-select popover), step4 (manager invites — role dropdown scoped to `MANAGER_INVITE_ROLES`, skippable), step5 (worker invites, skippable) → `completeOnboarding()` server action creates the `Organization` + `Facility` and sets the user's `organizationId`/`facilityId` → `/onboarding/complete` → "Go to Dashboard" → clean `/dashboard`, activation modal gone, sidebar now shows a "SETTINGS" section (Settings + Billing links, owner-only).

**Staff invites live in two equivalent places** using the same `InviteStaffModal` component + `createInvites` server action: (1) onboarding step4/5 (manager/worker invites during setup), and (2) post-onboarding at `/dashboard/settings` → "Users & Permissions" tab → "Invite user" button (also exposed at `/dashboard/staff`). The modal flow is: paste/type emails (or CSV import) → "Continue" → per-contact role assignment from a `groupRolesForSelect(inviterRole)`-scoped dropdown (owner sees all manager roles — Supervisor/HR/Clinical Director/Finance — plus all worker-category roles — Psychiatrist/Prescriber, Nurse, Therapist/Clinician, Case Manager, Behavioral Health Technician, Peer Support Specialist, Front Desk/Administrative Support, Facilities/Support Staff) → "Continue" sends the invite emails and shows an "Invite sent" success step. **Known minor bug:** the singular success copy renders "1 person hasbeen invited." (missing space) — cosmetic only.

**`/join/[token]` three-way branch** (confirmed still correct post-`origin/dev`-merge, matching the fix already verified in [[rbac-join-token-critical-bug]]): pending+unexpired → real join form (org name, role, invitee email pre-filled/disabled); `status='accepted'` → friendly "This invite has already been used" page with a "Go to login" link (HTTP 200, not 404); anything else (unknown token, expired-and-never-accepted) → genuine Next.js `notFound()` (HTTP 404). Completing the join form POSTs `/api/invite/accept` and redirects to `/login?joined=true`; the new user's role/org/facility exactly match the invite.

**Settings (`/dashboard/settings`) is owner-only**, enforced server-side (not just nav-hidden) — a non-owner hitting the URL directly gets a real access-denied page render ("You don't have access to Settings"), confirmed live with a supervisor account. Three tabs: **Users & Permissions** (default; team table + "Invite user"), **Roles** (read-only reference matrix), **Facility** (editable org/facility name + type via `updateFacility` server action — form uses react-hook-form `isDirty` to gate the Save button, shows a "Facility updated." success alert, and the new values persist across a hard reload; also shows current subscription plan read-only with a "Manage in Billing" link, out of scope for Stripe-free QA runs). There is **no separate "profile" tab inside Settings** — personal profile editing is a different route entirely, linked from the header user-menu dropdown ("Profile" → `/profile`, not `/dashboard/settings`).

**Minor a11y defect:** `InviteStaffModal`'s `DialogContent` uses a plain `<h2>` instead of the shadcn/ui `DialogTitle`/`DialogHeader` primitives (which onboarding's own CSV-upload dialogs use correctly) — triggers a Radix dev-console warning ("`DialogContent` requires a `DialogTitle`...") every time the modal opens. Non-blocking, but reproducible every run.

See [[local-dev-env-setup]] for the MailHog-must-be-started-manually gotcha that blocks this entire flow if skipped, and [[signup-test-patterns]] for the Radix-checkbox and server-action-mocking gotchas that still apply to the new `/signup` form.
