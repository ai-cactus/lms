**Local dev app location (2026-07-02 run):** `http://localhost:3000` (not :3005 as an older memory noted — port varies by session/branch setup). DB at `postgresql://postgres:••••@localhost:5433/lms`. Check `docker ps` for which of `lms-dev-db` / `lms-dev-redis` / `lms-dev-minio` are actually running — don't assume from `docker-compose.dev.yml` alone; only `lms-dev-db` was up by default in this run.

**Redis is required for the async PHI-scan/job queue and is easy to miss.** If Redis isn't running, uploads still "work" superficially (the modal doesn't obviously break) but the PHI scan silently fails with `ioredis Unhandled error event: ECONNREFUSED 127.0.0.1:6380` spamming the dev-server log. Fix: `docker compose -f docker-compose.dev.yml up -d redis` (self-serve, same category as starting the DB — do this without asking).

**Stripe webhooks have no listener in local dev (no `stripe` CLI installed, no `stripe listen` process).** This means any code path relying on `customer.subscription.created`/`.updated` webhooks to write to the DB (e.g. the *first* subscribe-from-nothing flow) will leave the DB never synced even though Stripe itself succeeded. Workaround (legitimate, not a bypass): use the `stripe` Node SDK (already a project dependency) to fetch the real Stripe object and construct a correctly-signed webhook event with `stripe.webhooks.generateTestHeaderString({ payload, secret: <real STRIPE_WEBHOOK_SECRET from .env> })`, then POST it directly to `http://localhost:3000/api/webhooks/stripe`. This exercises the app's real webhook handler code with real data and a real signature — a valid simulation of Stripe's own delivery. Note: **plan-swap/upgrade calls (`/api/billing/subscription/checkout` when a sub already exists) do NOT depend on the webhook** — they write to the DB synchronously in the same request, so this workaround is only needed for the very first subscribe.

**Vertex AI (Gemini) calls are likely to fail in this sandbox for TWO independent reasons — diagnose both before concluding "AI doesn't work here":**
1. **Network:** `fetch()` to `*.googleapis.com` from plain Node can fail with a bare `fetch failed` because this sandbox's IPv6 route is unreachable (`Network is unreachable`) while IPv4 works fine. Fix/diagnose by adding `--dns-result-order=ipv4first` to `NODE_OPTIONS` when starting the dev server (`next dev`). A quick way to test in isolation: `curl -sv --max-time 5 https://oauth2.googleapis.com/token` and watch for the IPv6-then-IPv4-fallback pattern in the verbose output.
2. **IAM/credentials:** `GoogleAuth()` (used by `src/lib/ai-client.ts` for `callVertexAI`) does **not** read the project's own `GCS_KEY_BASE64` env var — that's wired only into `src/lib/storage/gcs-provider.ts` for GCS. Vertex AI relies purely on Application Default Credentials. If ADC isn't configured, you'll see `Error: Could not load the default credentials.` Fix: decode the existing `GCS_KEY_BASE64` service-account JSON to a temp file and set `GOOGLE_APPLICATION_CREDENTIALS` to that path in `.env.local`, then restart the dev server. **But** — as of 2026-07-02, the service account behind `GCS_KEY_BASE64` (`<gcs-sa>@<project>.iam.gserviceaccount.com`) is storage-only and returns a genuine `403 PERMISSION_DENIED: aiplatform.endpoints.predict` once auth succeeds — confirmed via a direct out-of-band API call (`fetch` to `https://us-central1-aiplatform.googleapis.com/v1/projects/theraptly-lms/locations/us-central1/publishers/google/models/<model>:generateContent` with a minted token). **This means NO document/course generation can succeed end-to-end in this environment as of that date, regardless of content quality** — every AI call (PHI scan, course-detail auto-fill, course generation) fails identically. This is a real, fully-diagnosed environment/IAM gap, not a product bug — don't burn more time chasing it; test the failure-handling UX instead (it's fully exercisable and deterministic given this reliable failure) and clearly mark the AI-success-path criteria as BLOCKED (environment) in the report.
3. **Restarting the dev server safely:** Next.js 16 (Turbopack) runs a persistent dev-server daemon that can outlive the CLI invocation that started it (`.next/dev/logs/next-development.log` + a lock). `kill <old pid>` then re-running `next dev` from the repo root picks up a fresh `.env.local` correctly (confirmed: new PID, `Environments: .env.local, .env` in the startup line). The persistent-server's JSON log format **truncates structured logger object args to `{}`** — don't rely on it for detailed error inspection; instead tail the raw stdout redirect file from the `nohup ... > file.log 2>&1 &` you started it with, or reproduce the failing call standalone in a `node -e` script for full detail.

**Generating valid PDF/DOCX fixtures for the app's own `pdf-parse`-based extractor is pickier than expected.** Both `pdfkit` and `pdf-lib` (2026-07-02 versions) produced PDFs that Claude's own `Read` tool could open fine but that the app's `pdf-parse` v1.1.1 extractor rejected (`PDF Parsing Failed: bad XRef entry` / `Invalid PDF structure`). **`docx` (npm package) worked reliably** for DOCX fixtures with the same content — prefer DOCX over hand-generated PDF when a document just needs to reach the extraction step. See also [[document-hub-patterns]] for prior PDF-fixture notes (Node `pdfkit`) which are now superseded by this DOCX-preferred guidance for this specific extractor.

**Seeding data directly in the DB to route around AI-pipeline env gaps is a legitimate, project-sanctioned pattern** — the project's own `scripts/seed-test-user.js` does exactly this (inserts `Document`+`DocumentVersion` rows with inline `content`, bypassing upload/extraction entirely, to unblock downstream feature testing). Followed the same pattern for a `Course`/`CourseModule`/`Lesson`/`Quiz`/`Question` set (type `video`) to test the watch-through gate when neither AI course generation nor a video-course catalog were available — see [[watch-through-gate-testing]].

**MailHog is required for any email-gated flow (signup verification, invites, password reset) and is also NOT started by default**, even though `.env` already has `SMTP_HOST=localhost:1025` pre-configured pointing at it — confirmed 2026-07-09 on branch `rbac`. Unlike Redis (which fails open superficially), a missing MailHog makes `sendEmailVerification`/`sendInviteEmail` throw, and `signup()`/invite actions **fail closed**: the app deletes the just-created verification token and returns a user-facing error, so the entire signup or invite journey is un-runnable, not just missing a link to click. Fix: `docker compose -f docker-compose.dev.yml up -d mailhog` (self-serve, same category as Redis/MinIO — do this without asking). Once up, UI at `http://localhost:8025`, API at `http://localhost:8025/api/v2/messages` (list) — email bodies are quoted-printable encoded, so a link like `.../verify?token=abc` appears as `token=3Dabc` with soft line-break `=` chars; either decode properly or just cross-check the token against the DB (`verification_tokens`/`invites` tables) rather than regexing the raw body.

**MinIO is the storage fallback and is also NOT started by default** (`lms-dev-minio` container, port 9005/9006) — GCS was used successfully instead in this run since valid `GCS_KEY_BASE64` credentials were present and storage-scoped permissions worked fine (unlike Vertex AI). If GCS credentials are ever unavailable, start MinIO the same way as Redis (`docker compose -f docker-compose.dev.yml up -d minio`).

See [[watch-through-gate-testing]] for how a real playable video was sourced and seeded, and [[billing-subscription-patterns]] for the pre-existing Stripe test-mode UI notes (still accurate, complements the webhook-simulation note above).

**`.env.local` silently shadows a real `STRIPE_SECRET_KEY` in `.env` — always diff both before trusting "real Stripe credentials are configured" (2026-07-17).** Next.js loads `.env` then `.env.local` on top, and `.env.local` wins on any key present in both (confirmed via the dev server's own startup line `Environments: .env.local, .env`). In this run `.env` held a well-formed 107-char real Stripe test secret (`sk_test_51Tb...`) but `.env.local` held a 22-char placeholder (`sk_test_dummy...cal_qa`) that silently overrode it, so every server-side Stripe call (`stripe.prices.retrieve`, checkout, subscriptions) failed with `StripeAuthenticationError: Invalid API Key provided`, even though the task brief said real test-mode credentials were wired up. The app degrades this gracefully and correctly — plan cards show "Price unavailable" (`src/lib/billing-prices.ts`'s documented never-throw fallback) rather than crashing — so don't mistake that graceful-degradation UI for the bug itself; check the dev server log for `[billing] Failed to retrieve Stripe price` / `StripeAuthenticationError` to find the real root cause. **Before starting any real-Stripe billing QA run, verify with `node -e "require('dotenv').config({path:'.env.local'}); console.log(process.env.STRIPE_SECRET_KEY?.length, process.env.STRIPE_SECRET_KEY?.slice(0,12))"` (and the same for `.env`) that the *effective* key (the `.env.local` one, since it wins) is the real key, not a leftover stub** — then STOP and ask the user to fix `.env.local` if it isn't, rather than trying to work around a broken key. See [[billing-subscription-patterns]] for the Phase 4 run this blocked.

**The `next dev` process can die and silently self-restart mid-session (observed once, 2026-07-09, root cause not fully diagnosed).** Mid-QA-run, `curl http://localhost:3000` started returning connection-refused and an open `playwright-cli` browser session hit `chrome-error://chromewebdata/`. `ps aux`/`lsof -iTCP -sTCP:LISTEN` showed the original `next-server`/`next dev` processes gone, briefly replaced by a stray `next dev -p 3005` process, then within ~90 seconds a fresh plain `next dev` (defaulting to :3000) came up on its own and the app recovered cleanly with no manual restart. `.next/dev/logs/next-development.log` showed a routine `CredentialsSignin` auth error immediately before the gap (not obviously fatal) followed by relative-uptime-reset (`00:00:xx`) log lines confirming a real process bounce. **If you hit `chrome-error://chromewebdata/` or connection-refused against a previously-working `localhost:3000` mid-run, don't assume you broke it** — wait ~10-15s and retry `curl localhost:3000`/`lsof -iTCP -sTCP:LISTEN | grep 3000` before concluding the environment needs a manual restart; it may self-heal. If it does need a manual restart and the task said "don't restart," that instruction is about not restarting a *working* server — reviving an already-dead one to unblock testing is a different, reasonable call, but say so explicitly in the report.
