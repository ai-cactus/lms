description: next dev/Turbopack can omit custom @theme inline Tailwind utilities from the served CSS entirely, producing convincing-looking but fake styling "bugs" — always cross-check against npm run build && npm run start before reporting one
metadata:
  type: feedback
---

**Rule:** Before reporting a styling/CSS defect discovered under `npm run dev`, re-verify it against `npm run build && npm run start` (or at minimum fetch the served CSS chunk and grep for the specific utility class) before writing it up as a real product bug.

**Why:** During the `feat/partners` QA pass (2026-07-16), a fresh `npm run dev` session served CSS chunks that were **missing every custom marketing Tailwind token entirely** — `bg-surface`, `text-ink`, `bg-brand`, `text-brand`, `font-display` etc. (all newly added to `@theme inline` in `globals.css`) had zero occurrences in either served CSS chunk (confirmed by `curl`-ing the chunk URLs directly and grepping). This made the mobile nav menu render with a fully transparent panel (dark hero bled through, dark `text-ink` nav links nearly illegible against it) and the page background stay pure white instead of the intended warm `#f6f5f3` surface — a very convincing-looking regression. Re-running the exact same journey against `npm run build && npm run start` showed the identical CSS rules correctly compiled and the browser rendering pixel-correct. This was **not caused by a stale `.next` cache from a prior process** — it was the first compile of a freshly started `next dev` server against files that were already on disk. `rm -rf .next` was blocked by the permission system in that session, so the clean re-verification path was `npm run build && npm run start` instead, which is preferable anyway (matches [[e2e-local-verification-runbook]]'s existing CI-parity reasoning for e2e specs — the project has independently already moved away from `next dev` for a related route-group reliability issue).

**How to apply:** Whenever a styling gap is found under `next dev` (element looks unstyled/transparent/wrong-color, especially involving *newly added* custom Tailwind theme tokens or a route group not yet warmed up), treat it as unconfirmed. Fetch the actual served CSS (`curl -s http://localhost:3000/<route> | grep -oE '<link[^>]*\.css[^>]*>'`, then `curl` each chunk and `grep` for the specific class, e.g. `grep -c '\.bg-surface' chunk.css`) or just rebuild with `npm run build && npm run start` and redrive the same steps before writing up a defect. Only report it as a real product bug once it reproduces under the production build. If it only reproduces under `next dev`, report it as a dev-server/tooling observation (useful for the team to know), not as a product defect against the acceptance criteria.

See also [[marketing-partners-surface-patterns]] for the feature context this was found in, and [[local-dev-env-access]] / [[local-dev-env-setup]] for other dev-mode quirks in this repo (`next dev` can also silently self-restart mid-session).
