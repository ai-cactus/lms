**STATUS: FIXED AND RE-VERIFIED LIVE on 2026-07-06 (later same-day session).** `src/app/join/[token]/page.tsx` now does `const { token } = await params;`, guards blank/missing tokens with an explicit `notFound()`, and looks up the invite via `prisma.invite.findUnique({ where: { token } })` (token is `@unique`, so this can only ever match its own invite or none — no fallback query, no cross-invite/cross-org match possible even in principle). `src/app/api/invite/accept/route.ts` mirrors the same `findUnique`-by-token pattern for the actual account-creation POST. Re-verified with an adversarial two-pending-invite setup (deliberately left an *older* invite pending while accepting a *newer* invite's own link — the exact scenario that used to fail) — resolved correctly by token, not creation order; a fabricated/garbage token now cleanly 404s (HTTP 404, confirmed via `curl`). Full re-test evidence lives in `qa-reports/rbac-user-stories.md` (Story 4 section). **Still spot-check before fully trusting invite-link identity fidelity in a new session** — quick check: `grep -n "await params" "src/app/join/[token]/page.tsx"` and confirm `findUnique` (not `findFirst`) is used — regressions are always possible.

**Original bug write-up below, kept for historical context on what a regression would look like:**

**Found 2026-07-06 on branch `rbac` (six-role RBAC + Organization/Facility split), dev env (localhost:3000, Next.js 16.2.6).** `src/app/join/[token]/page.tsx` destructures `params` synchronously (`{ params }: { params: { token: string } }`) instead of awaiting it as a `Promise<{ token: string }>`, which Next.js 16 requires for dynamic App Router routes. This makes `params.token` evaluate to `undefined` at runtime. Prisma's `findFirst({ where: { token: undefined, status: 'pending' } })` treats an `undefined` field as "omit this filter" — so the query silently degrades to `findFirst({ where: { status: 'pending' } })` with **no `organizationId` scoping either** (confirmed by reading the fallback "already accepted vs 404" query in the same file, which also has no tenant filter).

**Observed, reproduced impact:** Visiting ANY `/join/[token]` URL — including a fully fabricated, nonexistent token — renders (and returns HTTP 200 for) whichever pending invite happens to be platform-wide "oldest" by insertion order. Submitting the account-creation form from that page actually creates/consumes THAT invite, not the one implied by the URL. Reproduced by: sending 4 pending invites (finance1/nurse1/frontdesk1/supervisor1), then visiting nurse1's own unique link and seeing finance1's email/role displayed; submitting the form there created finance1's account (with the name I'd typed for "nurse"), leaving nurse1's real invite untouched. Re-testing nurse1's link *after* finance1 became accepted then correctly showed nurse1 (since nurse1 had become the new oldest-pending) — proving the resolution always tracks "current oldest pending invite," not the URL.

**Practical QA workaround used to still get correctly-identified test accounts:** accept pending invites **in creation order** (oldest first) — each acceptance is coincidentally correct for whichever invite is currently platform-wide oldest. Do NOT trust that accepting invite N gives you invite N's account unless you've verified via DB (`SELECT email, role, status, token, created_at FROM invites ORDER BY created_at`) that it's currently the oldest pending one. After confirming account creation, immediately check `users`/`profiles` tables to confirm the email/role/name landed where expected before treating the acceptance as done.

**Blast radius:** Because the fallback query has no `organizationId` filter, this almost certainly also leaks invites **across different organizations** in a real multi-tenant deployment (any org's oldest pending invite would be served to any invitee's link, platform-wide) — this could not be empirically demonstrated in this session because the test DB only contained one organization, but the code path shows zero tenant boundary on this specific query.

**Do not assume this is fixed** in later sessions without re-verifying live — check `src/app/join/[token]/page.tsx` for whether `params` is now awaited (`const { token } = await params`) before relying on invite-link identity fidelity again. See [[worker-invite-join-flow]] for the earlier (different-manifestation) ambiguity bug on this same route.
