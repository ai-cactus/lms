**Confirmed live 2026-07-06 on branch `rbac`.** Categories: **manager** = owner (full incl. billing, never grantable via invite), supervisor (all minus billing), hr, clinical_director, finance. **worker** (8 roles, identical learner-only perms) = psychiatrist_prescriber, nurse, therapist_clinician, case_manager, behavioral_health_technician, peer_support_specialist, front_desk_admin, facilities_support.

**Invite-dialog role dropdown correctly enforces the grant matrix** (this part of RBAC works, in contrast to [[rbac-join-token-critical-bug]] which used to break the *acceptance* side — now fixed): owner/supervisor's "Add Staff" dropdown shows exactly 12 options (supervisor+hr+clinical_director+finance+8 workers, never owner). HR's dropdown shows exactly 11 options (hr+clinical_director+finance+8 workers, never supervisor/owner). **UPDATE 2026-07-06 (later session): Finance's "Add Staff" button/dialog is now FULLY HIDDEN, not just missing a role dropdown** — `StaffListClient.tsx` gates it behind `canInvite = can(dbRoleToRoleKey(inviterRole), 'invite.create')`, so finance (zero invite permission) sees no button/dialog at all. This closes the **frontend-affordance-vs-backend-permission gap** that used to exist here (previously: button/dialog shown, then a server-side "Forbidden" alert after submit). Same pattern was also fixed for supervisor's Billing nav item (see below) — this class of gap appears to have been addressed project-wide as of this fix set, but it's worth spot-checking any *new* role-gated UI for the same "affordance shown, backend silently rejects" anti-pattern going forward.

**Routing:** workers land on `/worker` (sidebar: Dashboard, Trainings, Certificates only — no staff/billing/course-authoring links at all, unlike the manager-nav gap above). Managers land on `/dashboard`. Navigating to a manager-only URL (`/dashboard`, `/dashboard/staff`, `/dashboard/billing`) while authenticated as a worker redirects all the way to `/login` (not back to `/worker`) — session remains valid the whole time (confirmed `/worker` still loads between attempts), so this is a hard route-group redirect, not a logout.

**Billing-scope nuance:** only **supervisor** is billing-restricted. **UPDATE 2026-07-06 (later session):** the Billing nav link is now correctly **hidden** for supervisor (`DashboardLayoutClient.tsx` gates it behind `can(role, 'billing.read')`), and direct navigation to `/dashboard/billing` now renders a proper styled access-denied card (`ShieldAlert` icon, "You don't have access to Billing" heading, back-to-dashboard link) via a server-side check in `src/app/dashboard/(main)/billing/page.tsx`, instead of the old raw unstyled "Forbidden" string. **HR and finance both have billing access** (page loads fine, finance even sees full billing UI with Change/Upgrade plan actions) — do not assume all non-owner managers lack billing.

**Self-signup does NOT create an organization immediately (non-obvious).** A brand-new owner signup only gets `role = owner` with `organization_id = NULL` — no organization row exists yet. The dashboard shows an un-dismissible `OrganizationActivationModal` ("Activate your account" → `/onboarding`) that can't be Escaped or clicked-outside-closed in welcome mode. You must complete the full 4-step onboarding wizard (`/onboarding/step1` org details → `step2` credentialing → `step3` services → `step4` invite-teammates, "Complete Onboarding" button) before the organization row is actually created and staff/billing/invite pages become usable. Budget for this when setting up a fresh owner account for testing — it's a real multi-field form (legal name, DBA, staff-count dropdown, contact info, phone, address, state dropdown, HIPAA-confirmation dropdown, business-type dropdowns, at least one program-service checkbox) and step 4's invite field can be left blank and skipped by clicking "Complete Onboarding" directly.

**Course-authoring is gated by ORG BILLING PLAN, not role** — see [[courses-billing-gate]]. In an org with no active paid plan, "Create Course" shows the same "A plan is required to create courses" paywall modal for every role, including owner. This means role-specific "cannot author courses" acceptance criteria CANNOT be verified in an unsubscribed test org — you'll always see the billing gate first regardless of role, and it will look like a false PASS for "restricted" roles and a false FAIL for roles that should be allowed. Needs a test org with an active (test-mode Stripe) subscription to properly isolate.

**Join-by-code:** self-serve join via a 6-digit code (generated at Profile → "Your Organization" tab → "Worker Onboarding" section → Generate Code, 6h expiry, reachable by any manager-category role incl. HR) always assigns the joiner role `front_desk_admin` (hardcoded as `DEFAULT_SELF_SERVE_WORKER_ROLE` in `src/lib/rbac/role-utils.ts`). The join flow requires the user to already be authenticated with NO organizationId (i.e., sign up via `/signup` → role-selection → "Worker" → verify email → log in → lands on `/worker` showing "Join your Organization" code-entry screen) — it is NOT a public/unauthenticated code-redemption page.

**Layout bug on the join-by-code confirmation screen — FIXED, re-verified 2026-07-06 (later session):** previously, at desktop viewport widths ≥1024px (Tailwind `lg` breakpoint, e.g. common 1200×769), a decorative right-hand hero image panel (`hidden lg:block`) visually overlapped and intercepted pointer-events on the "Join Organization" button (Playwright reported "subtree intercepts pointer events", requiring escalating retries or a resize below 1024px to click through). Re-tested at the exact same 1200×769 viewport after a defensive `z-10`/`z-0` stacking fix in `onboarding-worker` — the click now succeeds immediately with zero retries. Still worth a quick spot-check if this screen's layout is touched again.
