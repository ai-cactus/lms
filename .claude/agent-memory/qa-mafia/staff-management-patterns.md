**Location:** `/dashboard/staff` (sidebar "Staff Management"). Header shows "X/Y workers used • Z seats remaining (<plan name>)" — **this plan name/seat-count is subject to the same stale-display bug as Billing** (see [[billing-subscription-patterns]] ISSUE 3) — confirmed showing "(Starter)" even when the org was actually on an active Professional plan. Don't trust this seat-count as ground truth for which plan is active.

**Add staff — email only, no CSV here:** "Add Worker" button opens an "Invite New Staff" dialog with ONLY a chip-based email input (type email, Space/Enter to add) — confirmed via full DOM dump that there is no file-upload element anywhere in this dialog. Submits via `POST /dashboard/staff`, success shown inline as "Sent N invite(s)."

**CSV bulk-import does NOT exist in ongoing Staff Management.** The only CSV import feature in the whole product is the "Import with .csv file instead" button inside the ONE-TIME initial onboarding wizard (`/onboarding/step4` — see [[courses-billing-gate]] and Phase 1b history). That page remains reachable via direct URL post-activation, and its CSV parser works correctly in isolation (upload a `.csv`/`.xls` file, up to 10MB → shows "N emails found"). **DO NOT click "Complete Onboarding" from this page in a later session** — doing so resubmits the ENTIRE onboarding bundle (Org Details + Credentialing + Services + invites) in one request, and since only step 1 persists incrementally server-side (steps 2-3 are client-state only until final submit), reaching step4 via direct URL in a fresh session means steps 2-3's data is very likely empty/stale — submitting would risk **overwriting the org's real Credentialing/Services configuration with blank values**. Only the parse-and-preview step ("N emails found") is safe to test in isolation; treat completion as destructive.

**Row actions differ by staff status:** a **pending** invite's row-actions menu offers ONLY "Revoke Invite" (with a proper confirmation dialog: "Are you sure you want to revoke the pending invite for X? They will no longer be able to use the invite link to join your organization."). Pending rows are also NOT clickable — no detail page exists until the invite is accepted. An **active** staff member's row IS clickable, opening `/dashboard/staff/<uuid>` with a full profile: header (email + Job Title chip), action buttons **Edit Profile / Remove Staff / Assign Course**, 4 stat tiles (Total Courses Assigned, Courses Completed, Failed/Retake Needed, Active/Due Soon), and **Courses** / **Certificates Issued** tabs. Certificates tab shows a sensible empty state "No certificates available." pre-completion.

**DEFINITIVE FINDING — no functional staff/worker role distinction exists anywhere in this product.** Opening "Edit Profile" on an active staff member reveals the "Staff Member" label seen throughout the UI is simply the value of a free-text **"Job Title"** field (placeholder example: "e.g. Direct Support Professional") — NOT a role, permission tier, or account type. Combined with the Phase 1b finding (invites for "team members" and "workers" both land in one undifferentiated array with no role selector), this confirms: there is no admin-configurable role/permission tier distinguishing "worker" from "staff/team member" anywhere in the product — everyone invited is the same kind of record, differentiated only by this optional cosmetic string. If a story/objective asks to test role-based permissions for staff vs. workers, flag this up front — the feature doesn't appear to exist.

**Open question (unconfirmed, worth re-investigating):** two staff invited identically (same combined-array mechanism, same invite session) showed DIFFERENT statuses later — one had been assigned a course and showed as an active "Staff Member" with no pending indicator; the other (never assigned a course) still showed "Pending Invite". This suggests assigning a course to a pending invitee might silently convert/activate their record without them ever completing an actual invite-acceptance/signup flow — could not be confirmed via black-box testing alone (would need backend/DB access to check if e.g. a password/account genuinely exists for the "activated" one). Worth checking if this recurs.

**Assign Course dialog (from a staff profile):** "Assign Course" button → dialog "Assign Course to <email>" with an email chip (pre-filled) and a course-selection combobox. Correctly **disables itself** with "User has been assigned all available courses" when there's nothing new to assign — good defensive UX, confirmed accurate in testing (only one course existed org-wide, already assigned to the sole active staff member).

**Cross-screen status-label inconsistency (minor):** the same course enrollment showed Quiz Status "In Progress" on the worker's Staff Management profile, but "Not Started" on the course's own Enrolled Staff table (seen in [[courses-billing-gate]]'s Phase 4 testing) — reconcile before trusting either label in isolation.

Note: an older staging-environment memory ([[reminders-feature-patterns]]) mentions an "Assign manager" combobox on `/dashboard/staff/{userId}` — this was NOT observed on production during this Phase 5 pass. Could be a staging-only/unreleased feature, a different branch, or something missed — worth re ­checking if manager-assignment functionality is specifically in scope for a future phase.
