**Confirmed working (2026-07-02) on local dev — the watch-through gate is robust.** `src/lib/video/gating.ts` (pure logic: `clampSeekTarget`, `isQuizUnlocked`, 95% threshold, 1.5s forward-seek tolerance) + `VideoPlayer.tsx`'s `handleSeeking`/`handleTimeUpdate`/`handleEnded` correctly block scrub-to-unlock across every attack vector tried — see below. This is worth re-verifying rather than assuming stale if the fix is ever touched again, but as of this date it's solid.

**Seeding a real, playable video lesson without AI generation or a video-course catalog (both were unavailable — see [[local-dev-env-setup]]):**
1. Download a small, real, public-domain sample MP4 (e.g. `curl -sL -o sample.mp4 https://www.w3schools.com/html/mov_bbb.mp4` — ~788KB, ~10s Big Buck Bunny clip, confirmed a valid `ISO Media, MP4 v2` file via `file`). A tiny known-duration clip makes full-watch-through tests fast (~10s of real wait, not minutes).
2. Upload it directly to the org's real GCS bucket using the `@google-cloud/storage` SDK + the decoded `GCS_KEY_BASE64` credentials (storage-scoped creds work fine even when Vertex AI creds don't — see [[local-dev-env-setup]]), matching the app's own URI format: `gcs://${bucketName}/${key}` (confirmed via `src/lib/storage/gcs-provider.ts`).
3. Seed `Course` (type `'video'`, `status: 'published'`) → `CourseModule` → `Lesson` (with `video_storage_uri` = the `gcs://...` URI, `video_provider: 'self'`, `media_status: 'ready'`) → `Quiz` (FK'd to the lesson, `lesson_id` not `course_id`) → `Question` rows (separate table, NOT a JSON column on `Quiz` — `quizzes.questions` doesn't exist; questions relate via `quiz_id`). Assign via the normal admin "Assign" UI flow to create a real `Enrollment`.
4. The app's `/api/video/[lessonId]` same-origin proxy correctly resolved a signed URL from this real GCS object and streamed it to the `<video>` element — confirms the whole pipeline (not just the gating logic in isolation).

**How to reset watch progress between test variants (no UI reset exists):** the high-water mark is persisted directly on the `Enrollment` row (`video_position_seconds`, `progress`, `status`) via `saveVideoProgress` (`src/app/actions/video-progress.ts`) — NOT a separate video-progress table. Reset with `UPDATE enrollments SET video_position_seconds = 0, progress = 0, status = 'enrolled' WHERE id = '<id>'` then reload the `/learn/<courseId>` page.

**Rigorous seek-block test matrix (all passed 2026-07-02) — use `playwright-cli eval` directly on `document.querySelector('video')`, not UI drag-scrubbing (more precise, avoids flaky pointer-drag emulation):**
1. **Isolated large jump, no playback:** `v.currentTime = 9` (video paused, never played) → wait ~1s → confirm `v.currentTime` snapped back to the high-water mark (0 if untouched). **Do this test with the video PAUSED, not immediately after `.play()`** — calling `.play()` first and then checking state after any real wall-clock delay (including normal `playwright-cli` tool-call round-trip latency, which can easily be 1–3s per call) risks the short test video finishing playback for real in the background, giving a false "it unlocked" reading that has nothing to do with the seek being blocked. Isolate seek-clamp verification from playback entirely.
2. **Chained small-tolerance seeks (the subtle bypass to specifically check for):** repeatedly seek forward in steps just within the tolerance window (e.g. 1.4s when tolerance is 1.5s), each time waiting long enough for a `timeupdate` to fire, to see if the high-water mark can be "walked" forward via seeks alone (no real playback) all the way to unlock. In this codebase it correctly got stuck after 1-2 hops (didn't reliably keep advancing) and the quiz stayed locked — but this is exactly the kind of loophole worth re-checking on any future change to the tolerance/clamp logic, since the naive implementation (checking only the immediate delta) is theoretically exploitable if `handleTimeUpdate`'s guard ever weakens.
3. **Seek-near-end-then-let-it-finish (the most realistic real-world bypass attempt, and the one most worth testing every time):** seek to `duration - 0.3s`, confirm the clamp fires and resets to the high-water mark FIRST, then call `.play()` and confirm playback resumes from the clamped position (not the seek target) and does not reach `ended` quickly. This specifically guards against a naive `handleEnded` that unconditionally sets `maxWatched = duration` regardless of how the 'ended' event was reached — a real risk pattern given this codebase's `handleEnded` does exactly that unconditional assignment (by design, since reaching the end for real is only possible from genuinely-clamped positions when the seek-block works correctly — but this is the one place a future refactor could reintroduce a bypass).
4. **Full genuine playthrough:** let the (short, ~10s) video play natively start-to-finish; confirm `progress: 100`, `status: 'lessons_complete'` in the DB and "Proceed to Quiz" becomes enabled.
