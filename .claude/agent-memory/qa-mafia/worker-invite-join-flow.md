**FIXED as of 2026-07-02 (branch `fix/qa-report-001`, ticket THER-007) — re-verified live end-to-end.** `/join/[token]` now disambiguates its states: an **already-consumed** invite shows a friendly "This invite has already been used — An account has already been created with this invitation. Please log in to continue." page with a "Go to login" link (NOT a bare 404). A genuinely **unknown/garbage token** still correctly shows the generic "Page Not Found" 404 — the two are now visually and textually distinct. On the admin side, Staff Management now shows an explicit **"Expired" / "Expired Invite"** badge for expired invites (confirmed by manually setting `expires_at` to the past), and every pending/expired row's "Row actions" menu now offers **"Resend Invite"** and **"Copy invite link"** in addition to "Revoke Invite" — Resend mints a fresh token + future `expires_at` and the new link resolves correctly. Full write-up: `qa-reports/feat-qa-002-validation.md` (Story 5). The historical notes below (2026-07-01, pre-fix) are kept for context/regression reference; don't treat the "ambiguous 404" framing as current behavior.

**HISTORICAL — pre-fix state, confirmed 2026-07-01, RESOLVED 2026-07-02.**

**Route:** `/join/[token]` (`src/app/join/[token]/page.tsx`) — server component does `prisma.invite.findFirst({ where: { token, status: 'pending' } })`, joined with `organization`. Calls Next's `notFound()` (generic 404 page, not a custom "invalid/expired invite" screen) whenever the invite isn't found, isn't `status: 'pending'`, or `expiresAt` has passed. So a 404 on this route is **ambiguous by design** — it collapses "bad token", "already accepted", and "expired" into the same blank 404, with no distinguishing UI copy.

**Phase 6 blocker (2026-07-01):** the worker invite link `https://training.theraptly.com/join/596cf965-42c9-48aa-96fa-05c0e984e86e` (for `internovatechnologies+worker1@gmail.com` on org "LMS QA Test Org LLC") 404'd consistently (confirmed via reload, not a transient timeout — first `open` did time out at 60s but `goto` afterward loaded the 404 reliably twice). Follow-up login attempt with the specified email + the run's designated password (`QaWorker!Test2026$LMS`) returned the generic **"Invalid credentials."** — see [[production-env-access]] for why this message doesn't disambiguate "wrong password" vs "account doesn't exist/unverified". Net result: could not determine from the UI alone whether the invite was stale, already-accepted, or expired. Filed as BLOCKED in `qa-reports/phase-6-worker-side.md` and handed back to the orchestrator asking for a fresh invite link or the current invite/account status — did not guess or attempt to route around it.

**Takeaway for future worker-journey QA runs:** before starting a worker invite-acceptance test, confirm the invite is freshly issued (ideally the same session) rather than reusing a link from an earlier phase/handoff — invites appear to be single-use/status-gated (`status: 'pending'` required) and there's no visible expiry countdown or "resend" self-service on the 404 page itself (only generic "Go to Homepage" / "Back to Dashboard" links, the latter of which implies an admin-style page, not a worker one). If a provided invite link 404s, treat it as a hard blocker requiring a fresh link or DB-side confirmation — do not try alternate token guesses or bypass routes.

**IMPORTANT UPDATE (2026-07-01, same Phase 6 session): the "course-assignment invalidates a pending invite" theory was tested and NOT confirmed — a second invite (`internovatechnologies+staff1@gmail.com`, token `e2dc4542-0207-4838-89d6-3d3c409cbfe0`) that was explicitly NEVER assigned a course also 404'd, identically to worker1's (assigned) invite.** This is a control-case result: if assignment were the trigger, the never-assigned invite should have worked. Since it didn't, the more likely shared cause is **invite expiry** (both invites were issued in earlier phases, days before this Phase 6 run) or some other org-wide condition affecting all outstanding invites for "LMS QA Test Org LLC" — not something specific to the assign-course action.

**FURTHER UPDATE (same session): Staff Management (`/dashboard/staff`) is the reliable way to check an invite's REAL current status without DB access, and it revealed worker1 and staff1 are NOT in the same state:**
- **worker1** row shows **"Staff Member"** with no pending badge — the invite was **already accepted** (real account exists). Its 404 is therefore very likely *correct, expected behavior* for an already-consumed invite link, not a bug. The earlier "Invalid credentials." login failure is because the real password set during that actual acceptance is unknown (not necessarily the password named in a later test-run's instructions).
- **staff1** row shows **"Pending" / "Pending Invite"** — genuinely still pending per the same UI, yet its join link still 404s. **This is the real, still-unexplained anomaly**, isolated from worker1's (which has a mundane explanation). Row Actions menu on a pending-invite row only offers **"Revoke Invite"** — no resend/copy-link option to inspect or regenerate the live token non-destructively.

**Takeaway:** always check `/dashboard/staff` FIRST when a join link 404s — the row's status badge ("Staff Member" vs "Pending"/"Pending Invite") tells you in seconds whether the 404 is expected (already-accepted invite reused) or a real defect (still-pending invite that won't resolve). Don't treat every join-link 404 as the same failure mode.

**Do not assume "assign after accept" ordering alone will fix a 404'ing invite** — the real fix is very likely to use invites issued fresh, in the same test window they'll be accepted in, regardless of whether a course gets assigned before or after acceptance. If retesting, prioritize confirming actual `expiresAt` timestamps on the `invite` table (or the org's invite-expiry config) over reordering the assign/accept steps.

**Add Worker flow (admin side), confirmed working (2026-07-01):** Staff Management → **"Add Worker"** button → "Invite New Staff" dialog → type email into `textbox "Enter emails..."` → **Enter** to chip it → **"Send Invites"** button (disabled until at least one chip present). Fires `POST /dashboard/staff => 200` (observed firing multiple times back-to-back, likely a refetch/revalidate pattern, not multiple invites sent) and the new row appears immediately with "Pending Invite" / "just now".

**FINAL RESOLUTION (2026-07-01, same Phase 6 session): worker1's 404 is CONFIRMED as expected/correct behavior, not a bug.** Given worker1's real temp password (issued via the admin's course-assignment/onboarding path, not the original invite email), login at `/login` succeeded immediately with **no forced password-change step**, straight into `/worker`. This proves the account was already fully active — the original invite link 404s simply because it was already consumed. **staff1's "Pending but 404s" anomaly remains OPEN and unresolved** — filed as a real bug (see `qa-reports/phase-6-worker-side.md` ISSUE 1), needs dev/DB-side confirmation of that specific invite row.
